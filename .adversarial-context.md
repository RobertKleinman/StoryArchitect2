# Adversarial Review Context: Pipeline Re-Architecture

## What we're reviewing

A proposed **fundamental re-architecture** of Story Architect's pipeline flow. The current system is too slow, too expensive, and requires too much user interaction to produce output. We need honest assessment of: (1) whether the current architecture is salvageable or needs wholesale replacement, (2) whether the proposed new flow is sound, and (3) what specific design would work best.

Review broadly — don't limit to these concerns.

## Current Architecture (the problem)

Story Architect is a visual novel creation tool. The current pipeline has **6 sequential modules**, each with its own interaction loop:

1. **Hook** → clarifier Q&A (2-4 rounds) → tournament (3 candidates × builder + judge) → polish → summary
2. **Character** → clarifier Q&A (2-4 rounds) → tournament (3 candidates) → polish → summary
3. **Character Image** → clarifier → per-character builder + judge → summary
4. **World** → clarifier Q&A (2-4 rounds) → builder → judge → polish → summary
5. **Plot** → clarifier Q&A (2-4 rounds) → builder → judge → polish → summary
6. **Scene** → planner → per-scene (clarifier + builder + 2 judges) × 8-12 scenes

### The numbers
- **70-100+ LLM calls** per complete story
- **38 distinct LLM roles** with per-role model assignments
- **8-16 user interaction rounds** before any content is produced
- **6 sequential module handoffs** with user gates at each
- Multiple background services (psychology engine, cultural research, divergence explorer) adding more calls

### Existing valuable machinery
- **Psychology engine**: Tracks user behavior patterns (click vs type, deferral, confirmation rates) to adapt questioning
- **Cultural research service**: Tavily-powered background research for cultural specificity
- **MUST HONOR constraints**: User-confirmed facts propagated across all downstream modules
- **Judge/polish cycles**: Quality assessment and refinement
- **Divergence explorer**: Generates alternative creative directions
- **Story Bible**: Cross-module consistency checking
- **Comparative judging methodology**: Validated testing framework for evaluating output quality

### Why it's failing
1. **User experience**: Nobody wants to answer 12+ rounds of questions across 6 modules before seeing output
2. **Speed**: Sequential module execution with user gates = minutes to hours of wall clock time
3. **Cost**: 70-100+ API calls per story, many redundant (3-candidate tournaments, per-scene judges)
4. **Baseline comparison**: We built a test to compare app output vs. just chatting with Claude. The test itself times out because the pipeline takes too long to complete.

## Proposed New Flow

The architect (Claude, who built the current system) and the project owner propose collapsing to this:

### Step 1: Get the idea (1-2 user turns)
Single conversation to understand what story the user wants. Not module-by-module — just "tell me about your story." Uses psychology engine to adapt questions, but caps at 2-3 exchanges max.

### Step 2: Generate premise + synopsis (no user input)
System generates a complete premise and synopsis document. Runs cultural research, uses MUST HONOR constraints from Step 1. One generation pass, one judge pass.

### Step 3: User reviews premise (1-3 rounds max)
User sees the premise/synopsis and can suggest changes. Back-and-forth until satisfied. Hard cap at 3 rounds.

### Step 4: Generate story bible + scene descriptions (no user input)
Once premise is approved, system generates the full story bible (characters, world, plot) and descriptions of each scene. This is the heavy compute step but happens without user interaction. Judge/polish cycles run here but internally.

### Step 5: User reviews scenes (optional back-and-forth)
User sees scene outlines and can request changes to specific scenes. Limited rounds.

### Step 6: Full scene generation (no user input)
Approved scenes are generated as full prose/script. Per-scene generation with internal quality checks.

### Estimated improvement
- **~15-25 LLM calls** (down from 70-100+)
- **2-3 user interaction points** (down from 8-16)
- **2 user gates** instead of 6
- Most compute happens in batch without waiting for user

## Key questions for reviewers

1. **Is the current architecture salvageable** with incremental changes, or does it genuinely need this level of restructuring?
2. **Is the proposed 6-step flow the right design?** What's wrong with it? What's missing?
3. **Which existing services/machinery should be preserved** and which should be dropped?
4. **What are the risks** of this re-architecture? What could go wrong?
5. **How should the internal LLM orchestration work** within the batch steps (Steps 2, 4, 6)? Should it still use judge/polish cycles? Tournaments? Or single-pass with one quality check?
6. **What's the right interaction model** for the user review steps? How much control should the user have?
7. **Cost/quality tradeoff**: How do we maintain output quality while cutting 70-80% of LLM calls?
8. **Is there a better flow** than the one proposed?

## What the codebase looks like

- TypeScript monorepo (backend + frontend)
- Backend: Express routes per module, services per module, shared LLM client
- Frontend: React workshop components per module
- 38 LLM roles defined in shared/modelConfig.ts
- Pipeline runner script for automated testing
- Clarifier system with psychology tracking per module
- Cultural research via Tavily API
- Story Bible service for cross-module consistency

## Constraints
- Solo developer project — re-architecture effort must be practical, not academic
- The tool must justify its existence vs. "just chat with Claude" — speed AND quality matter
- Users are creative people, not engineers — the UX must be intuitive
- Budget: API costs per story should be reasonable for a consumer product
