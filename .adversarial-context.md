# Adversarial Review: Multi-Model Optimization + Prompt Simplification

## What we built/changed (2026-03-21)

### 1. Multi-model research pipeline
Changed background research roles from single-model (Claude Haiku) to multi-provider with diversity injection:
- `cultural_researcher`: Gemini Flash primary + GPT-5.4 Nano diversity (parallel)
- `grounding_researcher`: Gemini Flash primary + GPT-5.4 Nano diversity (parallel)
- `divergence_explorer`: GPT-5.4 Mini primary + GPT-5.4 Nano diversity (parallel)

Diversity calls run in parallel with primary, are non-fatal, and results are merged with deduplication by claim/reference prefix (first 60 chars lowercase).

### 2. Prompt simplification
Cultural researcher system prompt: ~1100 words → ~150 words
Grounding researcher system prompt: ~800 words → ~130 words
Divergence explorer prompt: UNCHANGED (A/B test showed no benefit)

Validated via comparative head-to-head judging: simplified wins 9-3.

### 3. Role-specific model assignments
31 LLM roles across 10 services, now using 6 different models:
- STRONG (Sonnet): 16 critical-path roles (clarifiers, builders, judges)
- FAST (Haiku): summaries, consolidator, summarizer, scene_divergence, world/plot polish
- GPT-5.4 Mini: premise_polish, divergence_explorer
- Grok 4.1 Non-Reasoning: char_polish
- Gemini Flash: cultural_researcher, grounding_researcher
- GPT-5.4 Nano: hook_escalation, cultural_summarizer (+ diversity injection)

### 4. Model catalog updates
Added GPT-5.4 Mini and GPT-5.4 Nano to PROVIDER_MODELS.
`detectProvider` already handles `gpt-` prefix.

### 5. OpenAI provider fix in test scripts
Test scripts use `max_completion_tokens` for OpenAI (not `max_tokens`).
Production `openaiProvider.ts` already handled this correctly.

## What we want reviewed
1. **Diversity merge logic** — Is the deduplication approach (first 60 chars of claim/reference) robust enough? Could it miss duplicates or wrongly deduplicate different items?
2. **Multi-provider reliability** — What happens when a provider is down? Are the non-fatal catch blocks sufficient? Could a slow diversity call block the primary?
3. **Model config complexity** — We went from 2 tiers (STRONG/FAST) to 6 different models. Is this maintainable? What breaks if a model is deprecated?
4. **Prompt simplification risks** — The simplified prompts dropped anti-fixation rules, search dimension definitions, early-project behavior guidance, and creative application mode definitions. Could these cause regressions on edge cases we didn't test?
5. **Schema compatibility** — Different providers handle schemas differently (OpenAI needs enforceAllRequired, Gemini needs stripAdditionalProperties). Is the production LLMClient handling all of these correctly for the new model assignments?
6. **Test infrastructure quality** — Are the test scripts and comparative judging methodology sound? Any systematic biases?
