### Remaining Objections

- **MAJOR**: Optional diversity calls still stall the pipeline
  - Architect said: They agree full detachment would be cleaner, but prefer keeping diversity on the awaited path because the merge must complete before caching; their proposed compromise is a 10 second timeout plus abort.
  - Still matters because: The remaining problem is less about infinite hang and more about systematic tail-latency inflation on cacheable reads. If the primary brief is ready quickly but cache population is delayed by an optional sidecar, then p95/p99 latency becomes governed by the diversity timeout window rather than the core generation path. That has second-order effects beyond UX: it ties worker occupancy, request concurrency, and cache warm-up speed to a non-essential branch. In practice this means transient slowness in Nano can reduce throughput of the whole feature even when the primary model is healthy. Abort fixes resource leakage after timeout, but it does not fix that the critical path is still waiting on optional enrichment.
  - Suggested resolution: Decouple cache correctness from diversity completeness. Cache the primary result immediately with a marker such as `diversityStatus: pending|skipped|complete`, then opportunistically patch in diversity only if it arrives within a very small merge window. If a second write is considered too much machinery, the compromise should at least reduce the timeout to a near-invisible budget and make the cache explicitly tolerant of a missing diversity payload.

### New Issues (raised this round)

- **MINOR**: The proposed nullable normalization strategy is correct in principle, but `set type to [T, 'null']` is not a complete generic transformation for all schema shapes. Optional fields expressed via `enum`, `const`, nested combinators, or schemas without a simple top-level `type` can be broken by a naive rewrite.
  - Fix: Implement nullability by preserving the original subschema and wrapping it generically, for example with `anyOf: [originalSchema, {"type":"null"}]` where needed, or by extending `enum` with `null` for enum-only fields. Add regression coverage across every structured-output schema reachable through OpenAI overrides, not just the currently known cultural/grounding cases.
