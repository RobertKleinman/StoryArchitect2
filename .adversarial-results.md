## CODEX — REJECT (severity: 0.84)

The production LLM layer is materially stronger than the blind-test harness, but the benchmark code is not measuring that production path. `scripts/research-blind-test.ts` performs provider-specific direct HTTP calls instead of going through `LLMClient`, does not validate or score schema compliance, and runs single-sample, non-reproducible comparisons. On top of that, model configuration in the app is a process-global mutable singleton, so evaluation changes can affect live traffic. The result is that the current methodology can produce misleading winner claims that mix true model quality with harness differences, transport quirks, and sampling noise. I did not find a `requirements.md` file in the repo, so there was no separate requirements document to diff against.

### Issues

- **CRITICAL** `scripts/research-blind-test.ts:228-349, 576-578` — The benchmark bypasses the production `LLMClient` stack and reimplements provider calls with raw `fetch()`. That means it does not exercise the real retry policy, timeout handling, truncation expansion/repair, provider adapters, or shared instrumentation used in production. The benchmark is therefore comparing models under a custom harness, not under the app's real execution path.
  - Fix: Refactor the script to call `LLMClient.call()` for each role/model, ideally by constructing an isolated client/config per run instead of issuing direct provider HTTP requests.

- **CRITICAL** `scripts/research-blind-test.ts:442-511, 576-640` — Schema-compliance failures are not actually detected or scored. Formatter functions treat malformed JSON as a display string like `(parse error) ...`, and missing arrays fall back to `?? []`, so partial or malformed outputs can still be shown to the human judge and potentially 'win' on readability. There is no pass/fail compliance metric, no required-field validation, and no tally of malformed responses.
  - Fix: Parse every response into typed objects, validate against the role schema, record structured failure reasons, and separate 'quality' scoring from 'schema compliance' scoring. Invalid outputs should not silently compete as normal entries.

- **CRITICAL** `backend/services/runtime.ts:25-67` — The app uses a singleton `llmClient` backed by a shared mutable `activeModelConfig`. Any config change affects the whole server process, not just one evaluation run or one request stream.
  - Fix: Make model config immutable at process level and pass per-request or per-job configs into a fresh `LLMClient` instance. Keep evaluation clients isolated from live app traffic.

- **CRITICAL** `backend/services/llmClient.ts:78-79` — `updateConfig()` mutates the live config object with `Object.assign()`. Combined with the singleton in runtime, this creates cross-request and cross-purpose contamination: benchmarking, admin model changes, and user traffic all share one mutable model assignment table.
  - Fix: Return new config objects instead of mutating the existing one, and avoid process-global mutable state for model selection.

- **CRITICAL** `backend/routes/models.ts:36-72` — The `/api/models` endpoint mutates the singleton `llmClient` for the entire process. If someone uses the UI or API to stage an evaluation, those changes immediately alter production behavior for concurrent users.
  - Fix: Scope model overrides to the request, session, or explicit evaluation job. Reserve global changes for privileged admin-only config with persistence and locking.

- **MAJOR** `scripts/research-blind-test.ts:240-325` — The benchmark introduces provider-specific request bias. Anthropic is tested through tool forcing (`tool_choice`/`tool_use`), Gemini through `responseSchema` after stripping `additionalProperties`, and OpenAI/Grok through strict `json_schema` response formatting. These are not semantically identical output contracts, so some observed quality differences can come from transport/protocol differences rather than model capability.
  - Fix: Normalize on the same internal provider abstraction used by production, and if provider-specific schema transforms are unavoidable, log them explicitly and treat them as part of the benchmark design rather than invisible plumbing.

- **MAJOR** `backend/services/providers/geminiProvider.ts:49-52` — Production Gemini requests pass `options.jsonSchema` directly as `responseSchema`, while the benchmark strips `additionalProperties` first because Gemini rejects it. That means the benchmark and production do not even test Gemini under the same schema shape, and production may fail for schemas the benchmark quietly repairs.
  - Fix: Move schema normalization into the shared Gemini provider so both production and evaluation use the same schema transformation path.

- **MAJOR** `backend/services/providers/anthropicProvider.ts:69-91` — Production Anthropic structured output uses `output_config`, while the benchmark uses tools and extracts `tool_use` input. Those are different API behaviors, so the benchmark is not representative for Anthropic even aside from the missing retry/truncation layer.
  - Fix: Use the same Anthropic provider implementation everywhere, including the blind-test script.

- **MAJOR** `shared/modelConfig.ts:12-16, 53-61, 128-151, 161-206` — The shared config correctly defines background roles like `divergence_explorer`, `cultural_researcher`, `hook_escalation`, and `grounding_researcher`, but the rest of the config surface does not fully honor that plan. The methodology says it compares background research roles, but the runtime management UI/API only partially supports them.
  - Fix: Make the config surface exhaustive and derive UI/API role lists from `ModelConfig` instead of maintaining separate hard-coded subsets.

- **MAJOR** `backend/routes/models.ts:9-21` — The runtime validator omits `hook_escalation` and `grounding_researcher` even though both exist in `ModelConfig`. As written, valid roles in shared config cannot be updated through the API.
  - Fix: Generate `MODEL_CONFIG_KEYS` from a single source of truth or at minimum add the missing roles and test for drift.

- **MAJOR** `frontend/components/ModelSettings.tsx:20-49, 71-116` — The mounted model settings UI only exposes Hook, Character, Character Image, and World roles. It omits Plot, Scene, divergence, cultural, grounding, and escalation roles, so the live configuration surface does not match the system's actual role set or the stated evaluation plan.
  - Fix: Either remove this incomplete admin UI or expand it so it fully covers the role matrix defined in `shared/modelConfig.ts`.

- **MAJOR** `backend/services/culturalResearchService.ts:392-447` — Grounding failures are swallowed and reduced to logs. If the grounding researcher errors or returns unparseable JSON, the service just proceeds without a cached grounding brief and without any reliability counter. That makes role-level reliability invisible in both production and evaluation.
  - Fix: Emit structured metrics for grounding call failures, parse failures, and empty-result cases, and surface them in debug/evaluation output.

- **MAJOR** `scripts/research-blind-test.ts:400-427, 430-434, 576-590` — The experiment has substantial uncontrolled variance. Each contract/role/model gets exactly one sample, temperatures are fixed but nonzero, and the blinding order uses unseeded `Math.random()`. Reported differences can therefore reflect sample noise and non-reproducible randomization rather than stable model quality.
  - Fix: Run multiple replicates per contract/role/model, record seed metadata, and use deterministic shuffling so the study is repeatable.

- **MINOR** `frontend/components/ModelSelector.tsx:1-132` — This component appears to be dead code in the current app path. `App.tsx` mounts `ModelSettings`, not `ModelSelector`. Keeping two competing model-config UIs increases drift risk, especially since they expose different role sets and editing models.
  - Fix: Remove the unused selector or converge on one model-settings surface.
